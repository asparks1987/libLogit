# libLogit v1.0.0-alpha.1 Release Notes

Status: draft release candidate notes.

v1.0.0-alpha.1 is the first working SDK milestone. It proves that a user can
add libLogit to a project, instantiate a `LOGIT`, emit logs, route output to
console and files, load named loggers from JSON config, and inspect a bounded
SQLite log store through SQL or the built-in viewer.

## Alpha Scope

The Alpha MVP language set is:

- Python
- C++
- C
- C#
- Java
- JavaScript
- Go
- Kotlin

Python is the reference implementation. Other bindings prove the shared object
contract and idiomatic language surfaces.

## Included Features

- Direct blank `LOGIT` construction.
- Structure-fed `LOGIT` construction.
- Config-loaded named `LOGIT` registries.
- Console, local file, and Alpha remote-path-as-file output.
- Text and JSON-lines formatting.
- Level filtering with `trace`, `debug`, `info`, `warn`, `error`, and `fatal`.
- Static metadata in JSON output where supported; Python also supports
  per-message metadata merging for JSON-lines and SQLite rows.
- Python SQLite `database` sink with bounded record-count, max-age, and logical
  payload-byte retention.
- Python file sink size-based rotation with numbered backups.
- Python redaction hooks for configured payload/metadata keys and message text
  patterns before console, file, network-file, or SQLite sinks receive events.
- Python async/batched buffering with capacity, interval, manual flush, and
  close-time flush behavior.
- Python per-event sink failure policies: warn, drop, raise, retry, and
  fallback file output.
- Python object-level thread-safety guard for configure/log/flush/close and
  concurrent file/buffered output coverage.
- Python reference package internals split into config, runtime, state,
  builder, redaction, formatter, event, level, and sink modules while
  preserving package-root imports.
- SQL-readable `logit_events` table.
- `liblogit-viewer` desktop window and CLI query mode.
- Shared conformance fixtures across the Alpha bindings, including fixed-clock
  metadata for timestamped golden fixtures.
- Package metadata for Python, C#, JavaScript, Java, Kotlin, and Go.
- Native C/C++ CMake install/export targets.

## Verification

The local release candidate gate is:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify-rc-local.ps1
```

That wrapper runs the full alpha verifier, confirms package metadata across the
MVP bindings, runs the JavaScript release artifact smoke by packing
`@liblogit/liblogit`, installing it into a fresh consumer, and verifying
installed-package logging, builds and inspects the C# NuGet artifact
`LibLogit.1.0.0-alpha.1.nupkg` from a temp copy, verifies the `.nuspec`,
README, and `lib/net10.0/LibLogit.dll` payload, builds and inspects the Java
Maven jar `liblogit-1.0.0-alpha.1.jar` from a temp copy, verifies runtime
classes and embedded Maven metadata, builds Python wheel/source-distribution
artifacts in a temp copy of the current tree,
inspects the generated artifact contents,
confirms the source distribution includes release docs, conformance fixtures,
and release verification scripts, verifies wheel/sdist filenames and internal
metadata, installs the wheel into an isolated virtual environment, and
smoke-tests installed-package logging to console, local file,
remote-path-as-file, SQLite, and the `liblogit-viewer` CLI. The wrapper also
checks the child alpha verifier exit code so a failed language matrix cannot
continue into artifact promotion.
For development-mode artifact re-checks, `-SkipAlphaMatrix` skips the full
language matrix but still runs the static release gates before package metadata
and release artifact checks.

After the release-candidate changes are committed, run the strict form on the
exact commit that will be pushed, released, or tagged:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify-rc-local.ps1 -RequireCleanWorktree
```

The strict form fails before building artifacts if local edits or untracked
files are present.

The underlying alpha verifier can also be run directly:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify-alpha.ps1
```

The local verifier runs Python tests, static checks with PowerShell and
JavaScript script syntax parsing, `ruff`, `mypy`, JSON Schema validation, and
native formatting, JavaScript tests, npm pack dry-run, installed npm tarball
smoke coverage with tarball and installed package metadata checks, C# tests and
NuGet pack, Go tests, Java tests and Maven install, Kotlin tests and Maven
package, C tests, C++ tests, native CMake build, CTest, install, and
installed-consumer checks. Native tool exit codes are checked explicitly so
failed package commands cannot produce a false green.

Hosted GitHub Actions uses `.github/workflows/alpha-matrix.yml` with the same
schema/config fixture coverage, PowerShell and JavaScript script syntax
parsing, static gate, package metadata sanity, Python wheel/source-distribution
inspection, and installed-wheel smoke coverage through
`scripts/verify-python-wheel-smoke.py`, plus package/build/test jobs. The
`scripts/check-hosted-ci.ps1` helper can pin the expected head SHA and verifies
each required Alpha job by name. Local static verification also checks that the
workflow job names and helper required-job list stay aligned, and that the
hosted Python wheel and JavaScript package smoke gates remain wired in. The
first hosted run after push still needs to be observed before the release notes
are promoted from draft to final.

## Migration Notes

Existing Python users can keep loading v0.1 configs through `init_from_config`
and the global `LOG(level)` helper. New code should use v0.2 registry configs
and named `LOGIT` objects.

See [Migrating From v0.1 Global Logging To v0.2 LOGIT Registries](../migration-v0.1-to-v0.2.md).

## Known Alpha Limitations

- Remote transports are not real network clients yet; non-socket remote paths
  are treated as file destinations in Alpha.
- Python is the only binding with the SQLite `database` sink in this release.
- Age-based file rotation, compression, cross-binding
  redaction/buffering/failure-policy parity, and richer backoff/fallback
  policies are beta-track.
- Hosted CI must still be observed after the current workflow is pushed.
- Java/Kotlin Maven Central publication polish, such as sources jars, javadocs,
  and signing, is release hardening work.
- The public API may still change before Beta based on Alpha usage.
