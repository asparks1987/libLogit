# libLogit v1.0.0-alpha.1 Release Notes

Status: draft release candidate notes.

v1.0.0-alpha.1 is the first working SDK milestone. It proves that a user can
add libLogit to a project, instantiate a `LOGIT`, emit logs, route output to
console and files, load named loggers from JSON config, and inspect a bounded
SQLite log store through SQL or the built-in viewer.

## Alpha Scope

The Alpha MVP language set is:

- Python
- C++
- C
- C#
- Java
- JavaScript
- Go
- Kotlin

Python is the reference implementation. Other bindings prove the shared object
contract and idiomatic language surfaces.

## Included Features

- Direct blank `LOGIT` construction.
- Structure-fed `LOGIT` construction.
- Config-loaded named `LOGIT` registries.
- Console, local file, and Alpha remote-path-as-file output.
- Text and JSON-lines formatting.
- Level filtering with `trace`, `debug`, `info`, `warn`, `error`, and `fatal`.
- Static metadata in JSON output where supported; Python also supports
  per-message metadata merging for JSON-lines and SQLite rows.
- Python SQLite `database` sink with bounded record-count, max-age, and logical
  payload-byte retention.
- Python file sink size-based rotation with numbered backups.
- Python redaction hooks for configured payload/metadata keys and message text
  patterns before console, file, network-file, or SQLite sinks receive events.
- Python async/batched buffering with capacity, interval, manual flush, and
  close-time flush behavior.
- Python per-event sink failure policies: warn, drop, raise, retry, and
  fallback file output.
- Python object-level thread-safety guard for configure/log/flush/close and
  concurrent file/buffered output coverage.
- SQL-readable `logit_events` table.
- `liblogit-viewer` desktop window and CLI query mode.
- Shared conformance fixtures across the Alpha bindings.
- Package metadata for Python, C#, JavaScript, Java, Kotlin, and Go.
- Native C/C++ CMake install/export targets.

## Install And Package Notes

Alpha artifacts are source-first until publication credentials and hosted CI
are finalized.

| Binding | Alpha Artifact |
|---------|----------------|
| Python | Source distribution and wheel from repository root. |
| C++ | Header-only API plus installed `libLogit::cpp` CMake target. |
| C | Static C target plus installed `libLogit::c` CMake target. |
| C# | `LibLogit.0.1.0-alpha.0.nupkg` from `dotnet pack`. |
| Java | `dev.liblogit:liblogit:0.1.0-alpha.0` jar from Maven. |
| Kotlin | `dev.liblogit:liblogit-kotlin:0.1.0-alpha.0` jar from Maven after installing the Java artifact. |
| JavaScript | `@liblogit/liblogit@0.1.0-alpha.0` package from npm pack. |
| Go | Go module under `languages/go`, intended for tag-based consumption. |

## Verification

The release candidate gate is:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify-alpha.ps1
```

The local verifier runs Python tests, static checks with `ruff`, `mypy`, JSON
Schema validation, and native formatting,
JavaScript tests and npm pack dry-run, C# tests and NuGet pack, Go tests, Java
tests and Maven install, Kotlin tests and Maven package, C tests, C++ tests,
native CMake build, CTest, install, and installed-consumer checks.

Hosted GitHub Actions uses `.github/workflows/alpha-matrix.yml` with the same
static gate plus package/build/test jobs. The first hosted run after push still
needs to be observed before the release notes are promoted from draft to final.

The promotion checklist is in
[v1.0.0-alpha.1 Release Candidate Checklist](v1.0.0-alpha.1-checklist.md).

## Migration Notes

Existing Python users can keep loading v0.1 configs through `init_from_config`
and the global `LOG(level)` helper. New code should use v0.2 registry configs
and named `LOGIT` objects.

See [Migrating From v0.1 Global Logging To v0.2 LOGIT Registries](../migration-v0.1-to-v0.2.md).

## Known Alpha Limitations

- Remote transports are not real network clients yet; non-socket remote paths
  are treated as file destinations in Alpha.
- Python is the only binding with the SQLite `database` sink in this release.
- Age-based file rotation, compression, cross-binding
  redaction/buffering/failure-policy parity, and richer backoff/fallback
  policies are beta-track.
- Hosted CI must still be observed after the current workflow is pushed.
- Java/Kotlin Maven Central publication polish, such as sources jars, javadocs,
  and signing, is release hardening work.
- The public API may still change before Beta based on Alpha usage.
